# Opsling — Technical Design

Companion to [`project-overview.md`](./project-overview.md). This document covers the **tech stack, project layout, key libraries, build/dev tooling, Docker setup, install flow, configuration reference, and release process**.

---

## 1. Tech stack

| Concern              | Choice                                     | Why                                                                 |
|----------------------|--------------------------------------------|---------------------------------------------------------------------|
| Runtime              | Node.js 22 LTS                             | Stable LTS, native `fetch`, native test runner available, broad VPS support |
| Language             | TypeScript 5.x (strict)                    | Type safety for a long-lived background process                     |
| Package manager      | pnpm                                       | Fast, deterministic, friendly to monorepo-ready layouts             |
| Dev runner           | `tsx`                                      | Run TS directly, fast watch mode                                    |
| Bundler              | `tsup` (esbuild under the hood)            | Single-step build to ESM, type-decls optional                       |
| Linter + formatter   | Biome                                      | One tool replaces ESLint + Prettier, fast, zero config drift        |
| Tests                | Vitest                                     | Fast, ESM-native, great DX; co-locates well with TS                 |
| Git hooks            | Lefthook                                   | Lightweight, language-agnostic, easy to opt out                     |
| Release / changelog  | Changesets                                 | Semver + autogenerated CHANGELOG                                    |
| Container image      | Distroless Node 22 (multi-stage Dockerfile)| Smaller image, smaller attack surface                               |

### Runtime libraries

| Library                  | Purpose                                                              |
|--------------------------|----------------------------------------------------------------------|
| `dockerode`              | Docker daemon client — events stream, stats, logs                    |
| `systeminformation`      | Host CPU, memory, disk, load                                         |
| `pino` + `pino-pretty`   | Structured logging; pretty in dev, JSON in prod                      |
| `chalk`                  | Colored startup banner                                               |
| `figlet` *(optional)*    | ASCII art for the banner; falls back to plain chalk if not desired   |
| `zod`                    | Strict `.env` parsing + runtime validation                           |
| `dotenv`                 | `.env` loading                                                       |
| `nanoid`                 | Stable IDs for incidents (used in logs and message threading)        |
| `p-queue`                | Tiny queue for Telegram send rate-limiting                           |

> No HTTP framework: the health endpoint uses Node's built-in `http` module. No Telegram SDK either — `POST` to the Telegram Bot API via native `fetch`. Keeps the dependency tree small.

### Why this layout

- **`src/collectors/`** and **`src/notifiers/`** are the two natural extension surfaces. Each lives behind a tiny interface (`Collector`, `Notifier`) so adding a new collector (e.g. `traefik` access-log scanner) or a new notifier (Slack, Discord) is a self-contained PR.
- **`src/core/`** has no I/O — it owns the alerting state machine. Easy to unit-test, no Docker required.
- **`src/lib/`** is leaf utilities — anything in here imports nothing from `core`/`collectors`/`notifiers`.
- **`src/app.ts`** is the composition root: it reads config and constructs everything. `index.ts` only handles process-level concerns (banner, signal handlers, top-level errors).
- **`docker/`** is intentionally separate from the repo root — install.sh and CI both fetch from there, and it keeps the root tidy.

## 3. Module responsibilities

### `config/env.ts`
- Loads `.env` via `dotenv`, validates with a `zod` schema, exits with a clear error message if invalid.
- Produces a frozen, typed `Config` object consumed by everything else. No code reads `process.env` directly.

### `core/alert-manager.ts`
- Receives `Reading { key, value, threshold, severity, meta }` events from collectors.
- Maintains per-key state: `OK | PENDING | FIRING | RECOVERED`.
- Applies:
  - `requireConsecutive` hysteresis on both directions.
  - `reNotifyAfter` cooldown while in `FIRING`.
  - Recovery delivery on `FIRING → RECOVERED`.
- Emits `NotificationEvent` to the notifier registry.
- Pure logic — fully unit-testable with a fake clock.

### `core/scheduler.ts`
- Owns the interval loop for poll-based collectors.
- Cancels cleanly on shutdown signal via `AbortController`.

### `core/health.ts`
- Tiny `http.createServer` listening on `HTTP_PORT`.
- `GET /health` → `200 { status, uptime, collectors: {...}, lastError }`.

### `collectors/docker/events.ts`
- Opens a single long-lived stream via `docker.getEvents()`.
- Translates `container start/stop/die/restart/oom/kill` into `Reading`s with severity hints (e.g. `die` with non-zero exit code → critical).
- Reconnects with exponential backoff on stream errors.

### `collectors/docker/logs.ts`
- For each watched container: `container.logs({ follow: true, stdout: true, stderr: true, tail: 0 })`.
- Demultiplexes Docker's log stream framing.
- Matches each line against `LOG_ERROR_PATTERNS` (and skips `LOG_IGNORE_PATTERNS`).
- Counts matches in a rolling `LOG_ALERT_WINDOW_SECONDS` window; emits a `Reading` when `>= LOG_ALERT_MIN_OCCURRENCES`.

### `collectors/system/*`
- Thin wrappers over `systeminformation`. Each file owns one metric.
- `disk.ts` runs on its own (slower) interval.

### `notifiers/telegram/formatter.ts`
- Pure functions: `(NotificationEvent) → string`.
- Renders MarkdownV2-escaped Telegram messages with severity emoji, host label, container name (when applicable), value, threshold, duration (for recovery), timestamp.
- All copy lives here — easy to tweak voice/tone without touching logic.

### `notifiers/telegram/queue.ts`
- Wraps sends in a `p-queue` with `intervalCap` aligned to Telegram limits (1 msg/sec/chat-ish, higher per-bot global).
- Retries with backoff on `429` / network errors; gives up after N attempts and logs.

### `lib/banner.ts`
```ts
// Printed once at startup, before pino is fully configured.
console.log(chalk.cyan.bold("  Opsling"));
console.log(chalk.gray(`  v${version} — watching your stuff so you don't have to`));
console.log();
```

### `lib/logger.ts`
- Single pino instance.
- `transport: { target: 'pino-pretty' }` when `NODE_ENV !== 'production'`, raw JSON otherwise.
- Levels: `debug` (verbose), `info` (lifecycle, incident open/close), `warn` (recoverable failures: telegram retry, docker reconnect), `error` (collector crashed, config invalid).

## 5. Docker

### `docker/Dockerfile` (multi-stage)

```
# 1. deps
FROM node:22-alpine AS deps
WORKDIR /app
COPY package.json pnpm-lock.yaml ./
RUN corepack enable && pnpm install --frozen-lockfile --prod=false

# 2. build
FROM node:22-alpine AS build
WORKDIR /app
COPY --from=deps /app/node_modules ./node_modules
COPY . .
RUN corepack enable && pnpm build

# 3. runtime (distroless)
FROM gcr.io/distroless/nodejs22-debian12 AS runtime
WORKDIR /app
ENV NODE_ENV=production
COPY --from=build /app/dist ./dist
COPY --from=build /app/node_modules ./node_modules
COPY package.json ./
USER nonroot
CMD ["dist/index.js"]
```

### `docker/docker-compose.yml` (what users actually run)

```yaml
services:
  opsling:
    image: ghcr.io/<owner>/opsling:latest
    container_name: opsling
    restart: unless-stopped
    env_file: .env
    environment:
      HOST_FS_ROOT: /host
    volumes:
      - /var/run/docker.sock:/var/run/docker.sock:ro
      - /:/host:ro,rslave        # host root, read-only; rslave propagates submounts
    ports:
      - "127.0.0.1:4717:4717"    # health endpoint, loopback-only by default
```

> - `docker.sock` is mounted **read-only**. Opsling only **reads** Docker state; it never starts/stops containers.
> - `/` is bind-mounted **read-only** at `/host`. This lets the disk collector see the host's real filesystems instead of the container's tiny overlay. Without it, disk alerts would report on the container, not the host.
> - The `rslave` propagation flag means new host mounts (e.g. an external drive plugged in later, or a Docker volume that becomes mounted) automatically appear under `/host/...` inside the container.
> - **macOS Docker Desktop note:** `rslave` is a Linux-only flag. Docker Desktop on macOS will accept `:ro` alone but the bind mount goes through the VM, so disk numbers reported will be the VM's, not your Mac's. For local dev on macOS, run natively with `pnpm dev` instead.

## 6. Install flow (`scripts/install.sh`)

What the script does, in order:

1. Verify `docker` and `docker compose` are available; if not, print install hints and exit.
2. Resolve the install layout based on UID:
   - **Root** → `/opt/opsling/` for files, `/usr/local/bin/opsling` for the helper CLI (FHS layout, matches what server admins expect to find).
   - **Non-root** → `$HOME/.opsling/` for files, `$HOME/.local/bin/opsling` for the helper CLI (works on Mac and Linux laptops without sudo).
   - Either can be overridden with `OPSLING_DIR=` and `OPSLING_CLI_DIR=`.
3. `mkdir -p <install-dir> && cd <install-dir>`.
4. If `.env` doesn't exist, download `.env.example` → `.env` and chmod 600. Otherwise leave the existing one alone.
5. Download `docker-compose.yml` next to it.
6. Apply any `--bot-token` / `--chat-id` / `--set KEY=VALUE` pre-seed values to `.env`.
7. Run the interactive wizard for any still-missing required values (TTY-aware, uses `/dev/tty` so prompts work under `curl | sh`).
8. `docker compose pull` (skippable with `--skip-pull` for local/offline testing).
9. Install the `opsling` helper CLI at `$OPSLING_CLI_DIR/opsling` (creates the dir if needed; warns if the dir isn't on `PATH`).
10. If both Telegram credentials are present, `docker compose up -d` automatically.

The script is **idempotent** — re-running it never overwrites a populated `.env`, and `--set` upserts cleanly. Safe to re-run to upgrade or reconfigure.

### Interactive vs piped behavior

`install.sh` detects whether it's running on a TTY (`[ -t 0 ]` with `/dev/tty` fallback so prompts still work under `curl | sh`):

- **Interactive**: prompts for `TELEGRAM_BOT_TOKEN` and `TELEGRAM_CHAT_ID` if they aren't already set in `.env`, writes them, then runs `docker compose up -d` automatically.
- **Non-interactive**: stops after preparing files and pulling the image. Prints next steps.

Flags:

| Flag                  | Effect                                                                 |
|-----------------------|------------------------------------------------------------------------|
| `--bot-token VALUE`   | Set `TELEGRAM_BOT_TOKEN` in `.env`                                     |
| `--chat-id VALUE`     | Set `TELEGRAM_CHAT_ID` in `.env`                                       |
| `--set KEY=VALUE`     | Set any env var in `.env` (repeatable — any `.env.example` key)        |
| `--skip-setup`        | Skip credential prompts entirely                                       |
| `--start`             | Force-start at the end (assumes `.env` is already filled)              |
| `--skip-cli`          | Don't install the `opsling` helper command                             |

Full pre-seed for automation:

```bash
curl -fsSL ... | sh -s -- \
  --bot-token xxx --chat-id yyy \
  --set CPU_THRESHOLD=90 --set IGNORE_CONTAINERS=opsling,traefik
```

When both Telegram credentials are present (from flags, prompts, or properly exported env vars), the installer auto-starts the container.

> **Env-var caveat.** `VAR=xxx curl ... | sh` passes `VAR` to `curl`, not to `sh`. The CLI flag path is the reliable one. Env-var pre-seed only works if you `export` the variable in the parent shell first.

### `opsling` helper CLI

The installer drops a small wrapper at `$OPSLING_CLI_DIR/opsling` (`/usr/local/bin/opsling` for root installs, `$HOME/.local/bin/opsling` for user installs). At runtime it **auto-detects the install dir** — prefers `$HOME/.opsling/docker-compose.yml`, falls back to `/opt/opsling/docker-compose.yml`. It's a thin sugar layer over `docker compose -f <install-dir>/docker-compose.yml ...`:

| Command           | What it runs                                              |
|-------------------|-----------------------------------------------------------|
| `opsling status`  | `docker compose ps`                                       |
| `opsling logs -f` | `docker compose logs -f`                                  |
| `opsling restart` | `docker compose restart`                                  |
| `opsling stop`    | `docker compose stop`                                     |
| `opsling start`   | `docker compose start`                                    |
| `opsling up`      | `docker compose up -d`                                    |
| `opsling down`    | `docker compose down`                                     |
| `opsling pull`    | `docker compose pull`                                     |
| `opsling update`  | `pull && up -d`                                           |
| `opsling config`  | open `.env` in `$EDITOR`                                  |
| `opsling health`  | `curl /health` (pretty-prints with `jq` if available)     |
| `opsling version` | print the image tag the compose file is using             |
| `opsling uninstall` | confirm, then `down` + `rm -rf $OPSLING_DIR`            |

> Hosting: the install URL (`https://opsling.dev/install.sh` or wherever) should serve the script from the repo's `main` branch, or pinned to a release tag. Pinning is recommended for v1 stability.

## 7. Running without Docker (advanced)

The supported install path is Docker, full stop. But Opsling is a plain Node.js process — if you really don't want the container, you can run it natively. **No installer covers this**; you do it by hand.

```bash
# clone + build
git clone https://github.com/go-native/opsling.git /opt/opsling-src
cd /opt/opsling-src
pnpm install
pnpm build

# config
mkdir -p /etc/opsling
cp .env.example /etc/opsling/.env
chmod 600 /etc/opsling/.env
$EDITOR /etc/opsling/.env

# systemd unit
sudo tee /etc/systemd/system/opsling.service >/dev/null <<'EOF'
[Unit]
Description=Opsling — Telegram-first monitoring watchdog
After=docker.service network-online.target
Wants=network-online.target

[Service]
Type=simple
EnvironmentFile=/etc/opsling/.env
WorkingDirectory=/opt/opsling-src
ExecStart=/usr/bin/node /opt/opsling-src/dist/index.js
Restart=on-failure
RestartSec=5
# Opsling reads the docker socket but never writes — adjust if you enable actions
SupplementaryGroups=docker

[Install]
WantedBy=multi-user.target
EOF

sudo systemctl daemon-reload
sudo systemctl enable --now opsling
```

What you lose with this path:

- **No image upgrades** — you `git pull && pnpm build && systemctl restart opsling` yourself.
- **No host bind mount** — you don't need `HOST_FS_ROOT`, since the disk collector now reads the native filesystem directly. Leave `HOST_FS_ROOT` unset in `.env`.
- **No `opsling` helper CLI** by default — its commands are `docker compose` aliases; with systemd use `systemctl status opsling`, `journalctl -u opsling -f`, etc.

If you find yourself wanting native install enough to script it, prefer adding a small **systemd unit generator** to the Docker install over forking the install model. Opening the Docker-vs-native matrix permanently doubles the maintenance surface.

## 7. Configuration reference

All values are read once at startup from `.env`. A change requires a container restart.

| Key                                 | Type / format               | Default            | Notes                                                                 |
|-------------------------------------|-----------------------------|--------------------|-----------------------------------------------------------------------|
| `TELEGRAM_BOT_TOKEN`                | string                      | *(required)*       | From BotFather                                                        |
| `TELEGRAM_CHAT_ID`                  | string / number             | *(required)*       | DM, group, or channel ID                                              |
| `HOSTNAME_LABEL`                    | string                      | `os.hostname()`    | Shown in messages — useful when running on multiple hosts             |
| `IGNORE_CONTAINERS`                 | csv or JSON array           | `opsling`          | Containers to exclude from monitoring. Default keeps Opsling out of its own alerting; set to empty to alert on self too. Everything not in this list is watched. |
| `CPU_THRESHOLD`                     | number (%)                  | `85`               | Host CPU                                                              |
| `MEMORY_THRESHOLD`                  | number (%)                  | `85`               | Host memory                                                           |
| `DISK_THRESHOLD`                    | number (%)                  | `90`               | Any mount over this fires                                             |
| `LOAD_THRESHOLD`                    | number (load avg / core)    | `4.0`              | Compared to 1-min load avg, normalized per core                       |
| `LOG_ERROR_PATTERNS`                | csv or JSON array of regex  | `ERROR,FATAL,panic,Exception,Traceback` | Case-insensitive                          |
| `LOG_IGNORE_PATTERNS`               | csv or JSON array of regex  | empty              | Lines matching these are skipped even if they hit an error pattern    |
| `LOG_ALERT_MIN_OCCURRENCES`         | int                         | `3`                | Per window                                                            |
| `LOG_ALERT_WINDOW_SECONDS`          | int                         | `60`               | Rolling window                                                        |
| `SYSTEM_INTERVAL_SECONDS`           | int                         | `30`               | CPU/mem/load poll cadence                                             |
| `DISK_INTERVAL_SECONDS`             | int                         | `300`              | Disk poll cadence                                                     |
| `REQUIRE_CONSECUTIVE`               | int                         | `2`                | Hysteresis depth                                                      |
| `RENOTIFY_AFTER_MINUTES`            | int                         | `30`               | `0` disables re-notify; one alert per incident only                   |
| `SEND_RECOVERY`                     | bool                        | `true`             |                                                                       |
| `DAILY_HEARTBEAT_AT`                | `HH:MM` 24h, or empty       | empty              | Local server time; empty disables                                     |
| `LOG_LEVEL`                         | pino level                  | `info`             |                                                                       |
| `HTTP_PORT`                         | int                         | `4717`             | `/health` endpoint                                                    |
| `HOST_FS_ROOT`                      | path or empty               | empty              | When set, disk collector reads the host's mounts via this prefix. Used in Docker deployments where the host root is bind-mounted (see §5). Leave empty for native runs. |

> **No per-container CPU/memory thresholds.** Opsling deliberately doesn't expose them. Container death/OOM events (via the Docker events stream) catch the "this container is broken" case more reliably, and host thresholds catch "the box is in trouble." Adding per-container resource alerts would just produce duplicate notifications when one container is consuming the whole host — better delivered as enrichment in the host alert (v2). See `docs/future_plans.md`.

## 8. Testing strategy

- **Unit (`tests/unit/`)** — covers the alert state machine, env parser, Telegram formatter, log pattern matcher. No I/O. Runs in CI on every push.
- **Integration (`tests/integration/`)** — spins up a throwaway `alpine` container and verifies the events collector sees its lifecycle. Requires Docker, skipped automatically if unavailable.
- **Manual smoke**:
  - `scripts/dev/seed-noisy-container.sh` runs a container that prints `ERROR` lines periodically and OOM-kills itself on demand — useful for end-to-end Telegram verification in dev.

## 9. Release process

- **Versioning**: SemVer via Changesets. PRs touching `src/` require a changeset.
- **Tagging**: merging a Changeset release PR cuts a `vX.Y.Z` tag.
- **CI on tag**:
  1. Build the multi-arch image (`linux/amd64`, `linux/arm64` — Hetzner has both).
  2. Push to `ghcr.io/<owner>/opsling:vX.Y.Z` and `:latest`.
  3. Create a GitHub Release with the autogenerated CHANGELOG entry.
- **install.sh pinning**: `install.sh` pulls `:latest` by default, but accepts `OPSLING_VERSION=vX.Y.Z` env var for pinning.

## 10. Operational concerns

- **Permissions**: read-only `docker.sock` mount. Runs as `nonroot` in the distroless image.
- **Memory footprint**: target < 80 MB resident under typical workloads. Verified in CI with a small load test.
- **Crashes**: the container `restart: unless-stopped` is the first safety net. The second is the optional external uptime checker against `:4717/health`.
- **Sensitive data**: never log `.env` values. Logger has a redact list for `TELEGRAM_BOT_TOKEN` and anything matching `*TOKEN*` / `*SECRET*`.
- **Time zones**: timestamps in messages are UTC by default; `TZ` env var (standard Linux) controls heartbeat scheduling and message formatting if the user prefers local time.

## 11. Open extension points (post-v1, but designed for)

- `Notifier` interface → drop in Slack / Discord / webhook with no `core` changes.
- `Collector` interface → add `nginx-access-log`, `traefik`, `journald`, etc.
- A future `enricher` stage between collector → alert-manager is where LLM-based summarization would slot in (see v2 ideas in the overview).
